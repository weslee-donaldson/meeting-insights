Threads: Cross-Meeting Client-Scoped Issue Tracking
Context
Common issues recur across multiple meetings within a client (e.g., "PR Deployments broken", "Production outages"). Currently there's no way to track these cross-meeting concerns. Users must manually scan meeting summaries to piece together the story.

Threads let users create named, client-scoped tracking items with a criteria prompt for LLM evaluation. Meetings are evaluated against threads using a two-step flow: embedding + FTS candidate preview → user-confirmed LLM classification. A dedicated nav view shows threads, their associated meetings (grouped/selectable, ordered by relevance), an LLM-generated summary (manually regenerated from selected meetings), and a persisted chat interface scoped to the thread context. Meetings display thread tags for cross-feature discoverability.

Design Decisions
Per-client scoped — each thread belongs to one client
Thread tags on meetings — each thread has a shorthand (max 10 chars) displayed as badges on meeting rows + MeetingDetail header
Single criteria_prompt — no separate keywords field; keywords go in the prompt
Two-step evaluation — preview candidates (embedding + FTS, free) → user confirms → LLM evaluates (costs tokens)
Append-mode re-evaluation — new associations appended by default; optional "override existing" checkbox re-evaluates previously associated meetings
Persisted chat — thread conversations stored in SQLite thread_messages table, survive app restarts
LanceDB chat context — embed user's question, KNN against associated meeting vectors, top-K most relevant meetings included (not all)
Manual summary regeneration — user clicks "Regenerate"; can select which meetings to include via inline checkboxes with group-by-day/series
Stale criteria indicator — when criteria_prompt changes, criteria_changed_at tracked; UI shows "stale" badge on associations evaluated before that timestamp with re-evaluate option
Meeting deletion resilience — FK cascade removes thread_meetings rows; thread_messages get context_stale flag + list of deleted meeting IDs
Meeting-side discoverability — thread tags on MeetingDetail header, "Part of Thread" sort on MeetingList

interface Thread {
  id: string;
  client_name: string;
  title: string;
  shorthand: string;            // max 10 chars, displayed as tag on meetings
  description: string;
  status: "open" | "resolved";
  summary: string;              // LLM-generated, manually regenerated
  criteria_prompt: string;      // single field — keywords go in the prompt
  criteria_changed_at: string;  // tracks when criteria last edited
  created_at: string;
  updated_at: string;
}

interface ThreadMeeting {
  thread_id: string;
  meeting_id: string;
  meeting_title: string;        // joined
  meeting_date: string;         // joined
  relevance_summary: string;
  relevance_score: number;      // 0-100
  evaluated_at: string;
}

interface ThreadMessage {
  id: string;                   // UUID
  thread_id: string;
  role: "user" | "assistant";
  content: string;
  sources: string | null;       // JSON array of source labels, nullable
  context_stale: boolean;       // set true when associated meetings deleted
  stale_details: string | null; // JSON: deleted meeting IDs/titles
  created_at: string;
}
Modified: core/db.ts — Add migration:


CREATE TABLE IF NOT EXISTS threads (
  id TEXT PRIMARY KEY,
  client_name TEXT NOT NULL,
  title TEXT NOT NULL,
  shorthand TEXT DEFAULT '',
  description TEXT DEFAULT '',
  status TEXT DEFAULT 'open',
  summary TEXT DEFAULT '',
  criteria_prompt TEXT DEFAULT '',
  criteria_changed_at TEXT NOT NULL,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL,
  FOREIGN KEY (client_name) REFERENCES clients(name)
);

CREATE TABLE IF NOT EXISTS thread_meetings (
  thread_id TEXT NOT NULL,
  meeting_id TEXT NOT NULL,
  relevance_summary TEXT DEFAULT '',
  relevance_score REAL DEFAULT 0,
  evaluated_at TEXT NOT NULL,
  PRIMARY KEY (thread_id, meeting_id),
  FOREIGN KEY (thread_id) REFERENCES threads(id),
  FOREIGN KEY (meeting_id) REFERENCES meetings(id)
);

CREATE TABLE IF NOT EXISTS thread_messages (
  id TEXT PRIMARY KEY,
  thread_id TEXT NOT NULL,
  role TEXT NOT NULL,
  content TEXT NOT NULL,
  sources TEXT,
  context_stale INTEGER DEFAULT 0,
  stale_details TEXT,
  created_at TEXT NOT NULL,
  FOREIGN KEY (thread_id) REFERENCES threads(id)
);
