1. Overview

Extend extraction output with a new field:

additional_notes (array of objects)

This field captures structured, logically grouped information from the meeting that is not already captured in existing artifact fields (summary, decisions, actions, questions, risks, features, technical topics).

Unlike other fields, additional_notes is meeting-adaptive:

The LLM determines the most useful structure per meeting

Objects may have different keys across meetings

Grouping and categorization should emerge from the meeting context

The system must remain robust and searchable despite dynamic structure.

2. Goals
Primary Goals

Capture valuable “extra” context and nuance not covered elsewhere

Encourage logical grouping, categorization, and nested structure where appropriate

Avoid duplication with existing fields

Preserve machine usability (store, embed, query, display)

Maintain strict JSON validity and ingestion resilience

Non-Goals

Enforcing a universal schema for notes

Guaranteeing identical keys across meetings

Introducing new DB tables or indexes in v1 (optional future)

3. Output Contract
3.1 Minimal contract

Extraction must always output:

additional_notes present

additional_notes is an array

Each element is a JSON object

JSON must be valid

Everything inside each object is up to the model.

3.2 Structural guidance (not schema)

The model should:

Prefer a small number of top-level group objects rather than many tiny notes

Use grouping keys such as category/theme/topic/section as appropriate

Use nested arrays/objects when it improves organization

Include enough detail to preserve reasoning and nuance

If there is nothing meaningful beyond existing fields, return [].

4. Extraction Behavior Requirements
4.1 Inclusion criteria

additional_notes must include information that is:

useful for later Q&A, recall, and context reconstruction

not already represented elsewhere in the artifact

Examples of good content:

rationale and tradeoffs behind decisions

unresolved tensions or disagreements (not necessarily “risks”)

constraints (org, legal, timeline, vendor, platform)

dependencies and sequencing considerations

customer context, motivations, politics, or stakeholder alignment

metrics discussed, definitions, measurement approach

notable quotes or strong viewpoints (sparingly, short)

historical context and “why now”

4.2 Exclusion criteria

The model must not:

restate the summary

re-list decisions or action items

turn open questions into notes (unless capturing deeper context around them)

create a “misc” dump of everything

4.3 Grouping requirement

The model should organize notes into logical groupings. For example:

“Architecture discussion” with subtopics

“Stakeholder concerns” with per-stakeholder breakdown

“Constraints and dependencies”

“Alternatives considered”

The actual grouping labels and keys are model-defined.

5. Storage Requirements
5.1 SQLite

Store additional_notes as JSON text in the artifacts row.

Column: additional_notes (TEXT/JSON)

Default for existing records: []

Migration must be idempotent

No requirement to normalize dynamic note objects in v1.

6. Embedding Requirements
6.1 Include in embedding input

The embedding input must include additional_notes content.

Because structure is dynamic, the system must canonicalize it into stable text:

deterministic ordering

consistent headings

avoid dumping raw JSON as-is without formatting

Goal:

maximize semantic retrievability

minimize noise

7. Query and Display Requirements
7.1 CLI output

--list summary and focused dumps should be extended with:

a new --list notes or --list additional_notes mode

output should preserve grouping, while remaining readable

7.2 Ask mode context builder

Ask mode should include additional_notes selectively to avoid context bloat:

include it after higher-signal fields

cap per-meeting notes size (by characters or tokens)

prefer “group headers + key points” format

8. Reliability and Validation
8.1 Validation

Validation rules for additional_notes must be minimal:

field exists

is array

all items are objects

JSON valid

No schema-level validation of internal keys.

8.2 Repair loop

If the artifact is invalid JSON or additional_notes is malformed:

attempt one repair retry

otherwise fallback:

set additional_notes=[]

continue ingestion

All failures must be logged to audit.

9. Observability

Track:

notes_count per meeting

approx notes_size (chars)

top-level grouping count (if detectable)

percent empty

repair usage rate

percent of meetings where notes exceed max cap and get truncated in context builder

This is needed because dynamic structure is harder to reason about.

10. Quality Constraints
Must

be additive, not redundant

be grouped and readable

preserve nuance and rationale where present

stay within practical size limits

Should

include stakeholder and constraint context when relevant

capture alternatives and tradeoffs

capture definitions and terminology when introduced

11. Risks and Mitigations
Risk: inconsistent structure makes downstream harder

Mitigation:

treat notes as “semi-structured”, store as JSON but canonicalize into text for embedding and display

Risk: notes become a dumping ground

Mitigation:

prompt emphasizes exclusion rules and grouping

enforce max size caps and log truncation rate

Risk: duplication with decisions/actions

Mitigation:

prompt explicitly forbids copying those items into notes

optional future: simple duplication detector (string similarity) to warn

12. Rollout Plan

Phase 1:

add storage column

update prompt to include additional_notes

add minimal validation + repair fallback

add --list additional_notes

Phase 2:

include canonicalized notes in embedding text

tune caps and formatting based on retrieval impact

Phase 3:

tighten prompt language based on observed failure modes
