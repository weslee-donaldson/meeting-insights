PRD: Meeting Intelligence Explorer — UI v1 (Final)
1. Product Model

This is a multi-column knowledge explorer with grounded chat.

It is not:

a dashboard

a task manager

a generative workspace

an insight engine

It is:

A structured meeting browser + raw context viewer + stateless grounded chat.

2. Layout Architecture
Global Structure
┌──────────────────────────────────────────────────────────────────────────────────────────────────────────┐

└──────────────┴───────────────────────┴──────────────────────────────┴────────────────────────────────────┘
3. Column Definitions
Column 1 — Clients

Purpose:

Hard context boundary

Single active client at a time

Behavior:

Selecting client:

Clears selection set

Refreshes meetings

Updates scope

Column 2 — Meetings (Grouped)

Purpose:

Display meeting instances

Allow multi-meeting selection

Features:

Grouped by series (collapsible)

Checkbox per meeting

“Select all in series”

Scrollable

Sorted newest first

Selection Rules:

Selection cannot cross clients

Date change prunes selection

No selection = whole scope

Column 3 — Context View

Purpose:

Render raw structured data

No intelligence layer

Behavior

If meetings selected:

Render selected meetings stacked

If none selected:

Render all meetings in scope stacked

Each meeting block:

Clear header (title, date)

Sections collapsed by default:

Summary

Decisions

Action Items

Open Questions

Risks

Proposed Features

Technical Topics

Additional Notes

No dedupe.
No aggregation.
No summary layer.

Collapsibility

Context column:

Collapsible

Resizable

If collapsed, chat expands to fill space

Column 4 — Chat

Purpose:

Stateless Q&A over current context

Core Rules

Chat context is derived from:

Selected meetings

Else → all meetings in client + date scope

Each question:

Stateless

No previous messages included

Only canonical context + current question

Chat UI

Minimalist:

Context: 3 meetings (TQ · 02/01–02/26)

[ Question input field                  ] [ Send ]

────────────────────────

Q: What decisions were made about OAuth?

A:
- Decision 1 …
- Decision 2 …

Sources:
- 02/26 DSU
- 02/25 Architecture

[Copy]

No avatars.
No bubbles.
No chat fluff.

Context Size Indicator

Near input:

Context: 3 meetings | 12,400 characters

Optional future:

token estimate

warning indicator

No hard cap enforcement in v1.

Context Change Behavior

When:

Client changes

Date range changes

Selection changes

Chat remains visible.
Next question uses updated context.
No system message appended.
No reset required.

Chat Column Collapsibility

Collapsible

Resizable

If collapsed, Context expands

4. Scope Bar

Always visible.

Controls:

Client dropdown

Date range picker

Reset button

Changing scope:

Refreshes meetings

Prunes selection

Updates context automatically

5. Interaction Matrix
Action	Result
Select client	Clears selection
Select meeting	Adds to selection
Clear selection	Context = full scope
Change date	Prunes selection
Collapse context	Chat expands
Collapse chat	Context expands
Ask question	Stateless LLM call

Minimalist

Relaxed spacing

Subtle 1px separators

No cards

No heavy shadows

Collapsible sections

Information-dense but breathable

Typography:

3-level hierarchy max

Clear section headers

Muted metadata

8. LLM Integration Rules

Provider:

Single provider at runtime

Config-based (local or anthropic)

System prompt:

Must restrict answers to provided context

Must require citations

Context packaging:

Deterministic section order

Deterministic meeting order (newest first)

Clearly labeled meeting blocks

9. What v1 Explicitly Does NOT Do

Deduplicate across meetings

Aggregate insights

Summarize multiple meetings

Persist chat memory

Allow cross-client context

Perform analytics

Limit context size artificially
