PRD: Local LLM Provider Integration (v1)
1. Overview

Add support for a local LLM provider to replace or complement Anthropic in:

Transcript artifact extraction

Ask-mode answer synthesis

The system must allow switching providers via configuration without modifying business logic.

The solution must:

Preserve the existing ingestion and query architecture

Maintain a single LLM boundary

Be resilient to malformed JSON from local models

Require minimal developer friction

2. Goals
Primary Goals

Introduce a provider-based LLM boundary

Support a local provider using Llama 3.1 8B Instruct via Ollama

Enable runtime switching between:

anthropic

local

stub

Maintain deterministic behavior in tests

Ensure ingestion never blocks due to malformed LLM output

Non-Goals

Rewriting the RAG system

Adding LangChain

Managing or embedding model weights directly

Supporting distributed LLM serving

3. System Architecture Changes
3.1 Provider-Based LLM Boundary

Replace the current “Anthropic adapter” concept with a provider interface.

The application interacts with:

LLM Provider
  ├── anthropic
  ├── local (ollama)
  └── stub

All LLM calls must pass through this boundary.

No other code may call Anthropic or Ollama directly.

3.2 LLM Capabilities

All LLM calls must specify a capability.

Initial capabilities:

extract_artifact

synthesize_answer

Future-compatible capabilities:

cluster_tags

generate_task

Capabilities are stable logical contracts, not tied to prompt filenames.

4. Local LLM Requirements
4.1 Runtime Model

Model:

Llama 3.1 8B Instruct (quantized acceptable)

Runtime:

Ollama

The system assumes Ollama is installed locally.

The application must NOT attempt to manage a daemon lifecycle directly.

4.2 Environment Configuration

New environment variables:

MTNINSIGHTS_LLM_PROVIDER=anthropic|local|stub
MTNINSIGHTS_LOCAL_BASE_URL=http://localhost:11434
MTNINSIGHTS_LOCAL_MODEL=llama3.1:8b

Behavior:

anthropic requires ANTHROPIC_API_KEY

local does not require ANTHROPIC_API_KEY

stub ignores all provider config

4.3 Startup Requirements (Local Mode)

When provider=local:

On startup, system must:

Verify Ollama server reachable

Fail fast with actionable error if not

During pnpm setup:

Verify model exists

Pull model if missing

Safe to re-run

During pnpm process and pnpm query:

Validate server reachable

Validate model available

Clear error if missing

No silent failures.

5. Extraction Requirements
5.1 Strict JSON Contract

extract_artifact must return valid JSON matching schema.

Behavior:

Validate schema

If invalid:

Retry once with repair prompt

If still invalid:

Store minimal artifact:

summary only

empty arrays for others

Log audit entry

Pipeline must never abort ingestion due to local model output.

6. Ask Mode Requirements
6.1 Deterministic Citations

The model must not invent sources.

Implementation rules:

Context blocks labeled [M1]…[Mn]

Model instructed to cite only those IDs

CLI prints Sources: derived from cited IDs

If no citations returned:

Print retrieved sources as fallback

Never print model-invented names

7. Observability
7.1 Logging

For each LLM call, log:

Provider

Capability

Model name

Latency

Token or character counts (if available)

Validation result (for extraction)

7.2 Audit Additions

Extraction audit must record:

Initial failure

Repair attempt result

Fallback usage

8. Evaluation Harness

Add lightweight evaluation runner:

Input: fixed question set

Output: JSONL results per provider

Fields:

question

retrieved meeting IDs

cited IDs

latency

answer length

Purpose:

Compare Anthropic vs Local

Compare model versions

No external eval platform required.

9. Failure Modes
Failure	Expected Behavior
Ollama not running	Fail fast with clear install/start instructions
Model not downloaded	Auto-pull during setup
JSON invalid	Retry once
JSON still invalid	Fallback minimal artifact
Model timeout	Surface error clearly
Ask produces no citations	Fallback to retrieved sources
10. Developer Experience

Expected workflow:

Anthropic mode
ANTHROPIC_API_KEY=...
MTNINSIGHTS_LLM_PROVIDER=anthropic
pnpm process
Local mode
MTNINSIGHTS_LLM_PROVIDER=local
pnpm setup
pnpm process

No code changes required to switch.

11. Success Criteria

Local provider considered production-ready when:

Ingestion completes reliably on >95% of transcripts without manual intervention

Ask mode produces grounded answers with correct source attribution

Switching providers requires only env change

Evaluation harness produces comparable metrics
